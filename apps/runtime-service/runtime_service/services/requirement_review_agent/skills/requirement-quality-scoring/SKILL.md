# 需求质量评分

使用本 skill 对需求质量进行维度化评分。

## 前置条件

必须先完成 `requirement-evidence-analysis`，且其输出中必须包含知识库证据基线（或"基线为空"的明确声明）。

## 评分维度

每个维度 0-20 分，总分 100 分：

- `business_objective`：业务目标和用户价值是否清楚
- `scope_boundary`：做什么、不做什么是否明确
- `workflow_and_rules`：主流程、分支、异常、业务规则是否完整
- `testability`：输入、输出、状态、验收标准是否可观察、可验证
- `risks_and_dependencies`：接口、数据、权限、边界条件、假设是否可识别

## 评分原则

- 只根据当前证据评分，不要脑补缺失业务规则。
- 需求越能直接支撑测试设计，分数越高。
- 关键规则缺失、状态不明、验收标准不可观察时要明显扣分。
- `review_score` 必须等于五个维度分之和。

## 知识库证据的使用

- **硬冲突红线**：需求内容与知识库中明确、现行有效的业务规则、接口约束或决策记录**直接矛盾**，且需求未说明变更依据的，属于硬冲突。存在至少一处经来源文档确认的硬冲突时，`workflow_and_rules` 不得高于 5 分，且必须同步压低其余受影响维度，**确保五个维度总分不超过 69**（即门禁必然为 blocked）。每处硬冲突都必须写入主要扣分依据，并引用来源文档中的具体规则原文。
- 判定硬冲突必须以检索到的来源文档为依据；若无法确认知识库规则是否仍然有效、或矛盾只是表述歧义，**不按硬冲突处理**，改列入待澄清项并正常扣分。
- 与知识库证据基线**矛盾**但未达到硬冲突标准的内容，在 `workflow_and_rules` 维度明显扣分，并将矛盾点写入主要扣分依据（注明来源文档）。
- 知识库中已有的接口约束、字段规则、状态流转在本轮需求中**未提及或未对齐**的，在 `risks_and_dependencies` 维度扣分。
- 需求内容与知识库证据**一致**时，可作为支撑高分的明确证据引用。
- 评分依据中引用知识库内容时必须注明来源文档；知识库证据基线为空时，不得以"与历史需求一致/冲突"作为加分或扣分理由。

## 输出要求

输出中间评分结果，不要给最终 JSON。

必须包含：

- `dimension_scores`
- `review_score`
- 主要扣分依据
- 可以支撑高分的明确证据

本 skill 不生成正式测试用例。
