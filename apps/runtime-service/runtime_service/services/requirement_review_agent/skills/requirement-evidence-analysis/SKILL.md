# 需求证据分析

使用本 skill 整理当前可用需求证据，为后续评分提供事实基础。

## 输入

- 用户输入的需求文本
- 当前轮上传 PDF/图片的多模态摘要或解析结果
- 用户明确给出的目标模块、业务规则、约束条件、验收标准
- 项目知识库中与本需求相关的历史需求、接口约束、业务规则（通过知识库工具获取）

## 知识库证据基线（必做步骤）

在整理本轮输入之后、输出中间分析之前，必须执行：

1. 从需求中提取核心业务实体、功能点、涉及模块，作为查询关键词。
2. 至少调用一次 `query_project_knowledge`，查询这些实体/功能点相关的历史需求、接口约束、业务规则、状态流转。
   查询时必须传 `only_need_context: true`——返回的是带来源的规则原文片段而非综述，
   便于逐条引用；一份来源文档里可能包含多条相关规则，逐段通读返回的片段，不要只看开头。
3. 将查询结果整理为"知识库证据基线"：逐条摘录规则**原文**并注明来源文档（引用返回的 citations），不要转述。
4. 对比本轮需求与知识库证据，标记：
   - 与历史需求或现有系统行为**矛盾**的点
   - 与已有功能**重复或重叠**的点
   - 知识库中已有约束但本轮需求**未提及**的点
5. 若知识库工具不可用或查询无结果，在输出中明确声明"知识库证据基线为空，后续结论仅基于本轮输入和上传材料"，不得虚构查询结果。

## 分析重点

- 业务目标和用户价值
- 功能范围和边界
- 主流程、分支流程、异常流程
- 业务规则、字段规则、状态流转、权限规则
- 可观察输入、输出、状态和验收标准
- 接口、数据、外部依赖、边界条件和假设

## 输出要求

输出中间分析，不要给最终门禁结论。

必须整理：

- `requirement_summary` 初稿
- 已明确的需求事实
- 知识库证据基线（含来源文档），或"基线为空"的明确声明
- 与知识库证据的矛盾点、重复点、未覆盖的已知约束
- 证据不足或存在歧义的点
- 评分时需要重点关注的风险

本 skill 不生成正式测试用例。
